1. Executive Summary & Business Context
Feature B4 — Document Management & Contract Lifecycle Management (CLM) — is the end-to-end document engine of the KT Onboarding Enterprise platform. It eliminates paper-based, email-driven, and manually rekeyed document processes by delivering template-driven generation, multi-party e-signature, OCR-powered ingestion, a secure per-onboardee document vault, and bulk-send capability — all natively within Salesforce, with no OmniStudio dependency.
💡  Strategic Positioning
B4 is a horizontal enabler for KT Onboarding. Every other feature — B5 (Identity Verification), B6 (Compliance Engine), B9 (Training & LMS), B13 (Personas) — depends on B4 to generate, store, and route documents. It is the document backbone of the entire platform.
1.1  Problem Being Solved
Without B4
With B4
Documents emailed as attachments; rekeyed manually into HR systems
200+ templates auto-generated with Salesforce merge fields — zero rekeying
No version control on offer letters, NDAs, or policy documents
Full document versioning with audit trail — every version stored in vault
Multi-party signing managed via email chains; no expiry enforcement
Automated signing order, delegation, expiry handling, and completion tracking
Uploaded passports and certificates parsed manually by HR
OCR auto-extracts structured data from scanned IDs and certificates
Bulk onboarding cohorts (100+ starters) took days to dispatch documents
Batch Apex processes 1,000+ simultaneous document sends asynchronously
Compliance audits required manual document assembly
Audit-ready document vault per onboardee — all docs versioned and timestamped
1.2  Key Value Propositions
200+ pre-built DOCX/PDF templates across all 8 KT domains — ready out-of-the-box
Dynamic generation using Salesforce data merge — no manual field population
KT Sign (native e-signature) included from Professional tier — no DocuSign license required for basic use
OCR ingestion via AWS Textract or Azure Form Recognizer — provider switchable via Custom Metadata

3. Architecture Overview
B4 is composed of five architectural layers: Document Generation, E-Signature Orchestration, OCR Ingestion, Document Vault, and Bulk Processing. Each layer is fully Salesforce-native, built on LWC + Apex + Salesforce Flow — with no OmniStudio components.
🏗️  Architecture Principle — No OmniStudio
All guided experiences previously handled by OmniScript are replaced with Salesforce Screen Flows (for step-by-step HR workflows) and LWC (for real-time interactive UI). All external API callouts previously handled by OmniStudio Integration Procedures are replaced with Apex HttpCallout classes using Named Credentials. This maintains full governor limit safety and upgrade compatibility.

5.1  KT_DocumentGeneratorService.cls — Core Service
This Apex class is the single entry point for all document generation. It accepts a KT_Onboarding__c ID, a KT_DocumentTemplate__c ID (or auto-resolves from CMT), merges Salesforce data, and produces a ContentVersion.
Method

Description

Primary generation method. Loads template binary from ContentDocument, executes merge field substitution, creates ContentVersion, creates KT_DocumentVault__c entry. Returns result wrapper with vault entry ID and ContentDocument ID.
generateFromDIE
generateFromDIE(Id onboardingId): List&lt;KT_DocGenResult&gt;
Auto-generates all documents required by DIE for the given onboarding record. Called by B1 DIE on record creation. Iterates KT_Document_Request__c records with Direction = Outbound.

Replaces all {{FieldPath}} tokens in template with real Salesforce field values. Supports related object traversal up to 3 levels (e.g., {{KT_Onboarding__r.Contact__r.Account.Name}}).

7. OCR Document Ingestion Pipeline

8. Document Vault — KT_DocumentVault__c
8.1  Vault Architecture
The document vault is a structured index layer on top of Salesforce Files (ContentDocument/ContentVersion). It provides enhanced metadata, access control, expiry tracking, and compliance tagging beyond native Salesforce Files capabilities.
Vault Capability
Implementation
Per-Onboardee Scoping
All KT_DocumentVault__c records are Master-Detail children of KT_Onboarding__c. Vault is automatically scoped to the onboarding record — no cross-onboardee document visibility.
Document Versioning
Each document re-upload creates a new ContentVersion linked to the same ContentDocument. Version_Number__c on KT_DocumentVault__c increments. Previous versions accessible via ContentVersion SOQL with IsLatest = false.
Access Control
KT_Admin and KT_HR_User can view all vault entries. KT_Manager can view documents for their onboardees only (sharing rules based on Onboarding__r.Hiring_Manager__c). KT_Onboardee (Experience Cloud user) can view only their own documents, filtered by Document_Type__c visibility rules in CMT.
Expiry Tracking
Is_Expired__c formula field drives a Scheduled Flow (KT_Document_Expiry_Alert) that creates tasks and sends email notifications 30/60/90 days before expiry. B6 Compliance Engine reads expiry status for compliance score calculation.
Audit Immutability
Shield Field Audit Trail configured on Status__c, Signature_Status__c, and OCR_Status__c fields with 10-year retention for regulated industries.
Virus Scanning
Uploaded documents scanned via configured security partner API (ClamAV-compatible endpoint) before ContentVersion is created. Files failing scan are quarantined (not deleted) and HR alerted.

9.3  Governor Limit Strategy
Governor Limit
Risk
Mitigation Strategy
Callout limit (100/transaction)
Document generation with external e-sign API calls per batch chunk
Batch size 25 records max per execute(); each record = max 2 callouts (generate + dispatch). Well within 100 limit.
CPU time (60s/transaction)
Large DOCX template binary processing and merge field resolution
Template binary loaded once per execute() from Static Resource or ContentDocument cache. Merge resolution uses indexed SOQL with LIMIT.
Heap size (12MB)
Large document binaries in memory during generation
Documents processed and written to ContentVersion one at a time within execute(). Batch size kept at 25 to prevent heap accumulation.
Async Apex queue (250 jobs)
Queueable chaining for large cohorts
Each batch chunk chains a single Queueable for post-processing. Batch completes all generation work; Queueable only handles lightweight status updates.
DML rows (10,000/transaction)
Creating KT_DocumentVault__c, KT_Document_Request__c updates, ContentVersion inserts
Batch size 25 × max 4 DML rows per record = 100 DML rows per execute(). Well within limits.

15.2  Feature Gate Enforcement
Feature gating is enforced via KT_Feature_Gate__mdt records checked in KT_DocumentGeneratorService.cls, KT_ESignatureOrchestrator.cls, and KT_OcrIngestionService.cls before executing any gated operation. If the subscriber org's tier does not include the requested feature, a KT_FeatureGateException is raised with an in-app upgrade prompt linking to the AppExchange checkout flow.

Appendix A — Glossary
Term
Definition
CLM
Contract Lifecycle Management — end-to-end management of document creation, negotiation, execution, and storage.
CMT
Custom Metadata Type — Salesforce metadata record used to store configuration data that is upgrade-safe and deployable via Metadata API.
ContentDocument / ContentVersion
Salesforce standard objects representing files. ContentDocument is the parent; ContentVersion is each version of the file.
KT Sign
KT Onboarding's native e-signature capability — canvas-based, legally valid, no third-party dependency.
Named Credential
Salesforce secure credential store for external API authentication — used for all B4 external callouts.
OCR
Optical Character Recognition — AI-powered extraction of structured text data from scanned document images.
Platform Event
Salesforce event-driven messaging mechanism — used in B4 for asynchronous document lifecycle event propagation.
Screen Flow
Salesforce declarative Flow type that renders user-facing guided experiences — replaces OmniScript in B4.
Shield Platform Encryption
Salesforce add-on that encrypts data at rest using customer-managed keys — applied to PII fields in B4.
Trusted Agent Identity
Salesforce Sprint 8 feature — adds mobile-push biometric authorization for high-value agent actions.
AXL
Agent Experience Layer — Salesforce technology that renders Agentforce agent interactions natively in Slack and Microsoft Teams.
